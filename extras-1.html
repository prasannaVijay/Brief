The silicon
under the model

GPUs, TPUs, and the CPU's comeback for agents.
matmul all the way down
Why care about chips?

The model is math. Where that math runs shapes everything.

Cost, latency, and what an agent can realistically do all trace back to which chip does the work — and there are three very different jobs.

Three jobs, three shapes: train the model, serve its tokens, and orchestrate the agent around it. Each one wants a different chip.

A short history

GPUs were built to draw pixels — then we noticed they could think.

In the '90s and 2000s, graphics cards did one thing brilliantly: the same simple math on millions of pixels at once.

NVIDIA's CUDA (2007) let anyone run general math on that hardware. Then in 2012 a neural net trained on GPUs — AlexNet — crushed the field, and the whole AI era has run on graphics cards ever since.

The core idea

A few big brains, or a thousand small ones.

CPU
few powerful cores · sequential, branchy
GPU
thousands of simple cores · same op, in parallel

A CPU is built to make hard decisions fast, one after another. A GPU is built to do the same operation on huge piles of numbers all at once.

The match

A matrix multiply is the same step, a million times over.

Multiplying matrices is thousands of independent multiply-and-add operations — none waiting on any other. That's exactly the shape a thousand parallel cores devour.

Which is why a GPU multiplies big matrices orders of magnitude faster than a CPU ever could.

Why this is an LLM story

Inside the model, everything is a vector.

token → (a vector)

Tokens become vectors. Attention — from the pre-read — compares those vectors with dot products. Stack the comparisons up and it's one big matrix multiply.

The model is matrix math wearing a language costume.

Even retrieval

Every "search" is multiplication too.

Semantic search and RAG compare your query vector against thousands of stored vectors. Each comparison is a dot product; all of them together are a single matrix-by-vector multiply.

So the whole stack — generating and searching — is matrix math. That's the work GPUs were accidentally perfect for.

Where GPUs are unbeatable

Training is a GPU's dream job.

Enormous matrix multiplies over huge batches, every core kept busy, pure throughput — the problem is embarrassingly parallel.

For training, nothing touches a room full of GPUs (or TPUs). This is their home turf, no argument.

The twist

But generation doesn't fill the GPU.

Inference makes one token at a time, each depending on the last, often for a single user. Now the bottleneck isn't compute — it's memory bandwidth: you stream the weights and the ever-growing KV cache for every single token while most cores sit idle.

Compute-rich, memory-starved. (Batch many users together and GPUs get efficient again — which is why they still serve most inference today — but the fit is worse, and that opens a door.)

Purpose-built

TPUs: chips built for tensors, not pixels.

A GPU is a graphics chip repurposed for AI. A TPU is designed from scratch for the one job that matters — multiplying tensors — using a layout that keeps data flowing with far less memory traffic.

The payoff is efficiency: more useful work per watt and per dollar on ML math, at both training and serving scale. Specialization beats leftover generality.

The comeback

An agent is mostly orchestration — and that's CPU work.

Most of an agent's time isn't matmul. It's calling tools, waiting on APIs and databases, branching, parsing, retrying — sequential, latency-bound logic. Exactly what CPUs are best at.

The heavy math is a remote model call; the agent's brain — the loop itself — runs happily on a plain CPU. As the work shifts from training models to running agents, the humble CPU is back on the critical path.

The one slide to keep

Right chip, right shape of work.

Train the model
giant parallel matmul
GPU / TPU
Serve the tokens
memory-bandwidth bound
TPU · inference chips · batched GPUs
Run the agent
orchestration + I/O
CPU

The skill isn't picking "the fastest chip" — it's matching the chip to the shape of the work.
