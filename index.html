Why this exists

One idea has to be solid before we build anything.

Agents, RAG, MCP, the router — everything in the series sits on top of how a language model actually works. Get this, and the rest is plumbing.

Already fluent in this? Skim it — it's the shared vocabulary we'll use every week.

The whole trick

A language model does one thing: it predicts the next token.

hello, how are → ?

Summarizing, drafting, answering, reasoning — all of it is this single step, run over and over.

Step one · tokenize

First, text becomes tokens.

The model never sees letters. Your text is chopped into tokens — whole words or word-pieces — each mapped to a number.

hello15339 ,11 how1268 are527

Those IDs are illustrative — but everything downstream is just math on numbers like these.

Step two · predict

It outputs a probability for every possible next token.

hello, how are

Not one answer — a ranked distribution over the whole vocabulary. Here are the top few.

unlikelylikely
you92%
things3%
we2%
ya1.5%
……
It's a dice roll

The model samples from that distribution — it doesn't "know" the answer.

Temperature sets how adventurous the roll is. Low temperature almost always picks you; high temperature lets unlikely tokens win sometimes.

This is why the same prompt can give different answers — and why "be more creative" and "be more reliable" pull the same dial in opposite directions.

Step three · repeat

Append the token, feed it all back, predict again.

hello, how are→you
hello, how are you→doing
hello, how are you doing→today
hello, how are you doing today

This loop — predict, append, repeat — is called autoregression. A whole paragraph is just this step a few hundred times.

A consequence that matters

Each call is independent. The model has no memory.

Between two calls it remembers nothing. What feels like memory is just the whole conversation being re-sent every single time.

Hold onto this — when we build Brief, we are responsible for state: config, history, what's already been sent.

The model's field of view

It can only use what fits in the context window.

Everything the model reasons over — the prompt, the history, any fetched data — has to fit in a fixed budget of tokens. Outside it, the text simply isn't there.

A wiki or a year of notes won't fit. That gap is exactly the problem RAG solves later in the series.

Optional · under the hood

How does it "know" to say you? It weighs the earlier words.

Predicting after are, the model scores how much each earlier token matters — this mechanism is called attention:

hello18%
,9%
how32%
are41%

You don't need the math. The takeaway: the model leans hardest on the most relevant words — here, "how" and "are" — to shape what comes next.

Know the edges

Brilliant at language. Unreliable at facts.

Lean on this

  • Summarizing, rewriting, reformatting
  • Drafting and explaining
  • Classifying and extracting structure
  • Following patterns in the prompt

Don't trust this

  • Facts outside training (it invents — "hallucinates")
  • Anything real-time or recent
  • True memory across sessions
  • Exact math and counting

Design rule for the whole series: use the model for language, and supply facts through tools and retrieval — never assume it knows.

Where we're headed

Brief is this predictor, plus tools, a loop, and state.

Every session adds one capability around this core: tools to fetch, a loop to act, RAG for knowledge, MCP for connections, skills to package, a router to choose.

Pocket glossary

Six words we'll keep using.

token
A word or word-piece — the unit the model reads and writes.
distribution
The model's ranked guesses for the next token, with probabilities.
temperature
How adventurous the sampling is. Low = safe, high = varied.
autoregression
Predict, append, repeat — how full responses get generated.
context window
The fixed token budget the model can see at once.
hallucination
A confident answer that's simply made up.
